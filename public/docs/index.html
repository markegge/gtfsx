GTFS Studio Documentation

A walkthrough of every part of GTFS Studio, ordered the way you'll actually use them. If you've never authored a GTFS feed before, start with What is GTFS? first, then come back here.

1. Agency setup

Every GTFS feed needs at least one agency record (agency.txt). In the Agency panel, enter your agency name, public-facing URL, timezone, and optional contact details. Multiple agencies are supported for joint feeds (for example, a city operator plus a regional partner) — each route is then assigned to a specific agency.

2. Service calendars

Service calendars (calendar.txt) define which days of the week service runs and over what date range. Toggle the day-of-week boxes for each service pattern (typical patterns include Weekday, Saturday, Sunday) and set a start and end date. For holidays and one-off changes, add exception dates (calendar_dates.txt) — you can either add service on a date that wouldn't normally have it, or remove service for a holiday. The bulk-add tool covers common US federal holidays in one click.

3. Routes & shapes

Routes are defined before stops in this tool. The reasoning: alignment is the single most consequential design decision for a transit service, so it should be locked in before you start placing stops or building schedules. To create a route, click "New route", fill in the short name, long name, color, and route type (bus, rail, ferry, etc.), then draw the route on the map.

Drawing modes

  • Snap-to-road (default): your line follows the road network using the Mapbox Map Matching API. Best for fixed-route bus and most rail.
  • Freehand: draw exactly where you click. Useful for off-road segments, ferry routes, or sketching alignments that don't yet have a road.

Edit shapes by dragging vertices, inserting new points, or deleting points. Multiple shape variants per route are supported — typically you'll want one for each direction of travel.

4. Stops

With a route selected, click along the route line to drop stops. By default each stop snaps to the nearest point on the route line and is rendered visually offset to the right side of the line, mimicking real-world curbside placement. To place a stop that isn't on any route (a park & ride, a transfer center, an off-street terminal), toggle freehand stop mode.

If you click near an existing stop, GTFS Studio will prompt you to reuse that stop instead of creating a duplicate. This is the right behavior — a stop served by multiple routes should be a single record in stops.txt.

Reorder stops along a route by dragging them in the sidebar list. Stop attributes (name, code, wheelchair boarding, parent station) are edited in the property panel.

5. Timetables & trips

The timetable editor is a grid: each row is a trip, each column is a stop, and each cell is an arrival/departure time. Create a trip, set its headsign and direction, assign it to a service pattern, then enter the time at the first stop. Intermediate stop times can be auto-interpolated from the route geometry and an assumed average speed, then refined per cell.

For repeating service, use "Duplicate trip with offset" to spawn a new trip every N minutes — much faster than typing a full weekday schedule by hand. For headway-based service (every 10 minutes from 6am to 9pm, no scheduled times), use the frequency editor instead, which writes frequencies.txt.

6. Fares

Fares are strongly encouraged. Trip planners and rider-facing apps depend on accurate fare data, and a feed without fares is materially less useful for the people consuming it. GTFS Studio will surface a persistent reminder if no fare data is defined.

The simplest setup is a single flat fare assigned to all routes. For zone-based systems, define fare zones on the map, assign stops to zones, and create a zone-to-zone fare matrix. Multiple fare types (regular, reduced, senior, student) are supported via separate fare_attributes.txt records.

7. GTFS-Flex zones & booking rules

For demand-responsive service (microtransit, dial-a-ride, deviated fixed-route), draw polygon zones on the map for each flex service area. Each zone gets pickup and drop-off windows instead of fixed stop times, plus an associated set of booking rules: how far in advance riders must book, what the booking phone number or URL is, what message text to display in trip planners, and so on. See the GTFS-Flex overview for more on what this spec extension covers and when to use it.

8. Validation

Validation runs continuously as you edit — errors and warnings appear inline next to the affected entity, and a summary panel lists everything outstanding. Errors block export (these are spec violations that would make the feed unusable). Warnings allow export but are flagged for your attention (for example: a route with no fare assigned, or a service pattern with no holiday exceptions). Click any validation message to jump directly to the editor section that owns it.

9. Export & import

Export produces a standards-compliant GTFS ZIP containing all required files plus any optional files that have data. Empty optional files are not exported. Import accepts any GTFS ZIP — required files are parsed into the editor; non-standard files are preserved on round-trip but flagged in the import summary.

10. Coverage & demographic analysis

The coverage panel overlays US Census ACS data on the map and reports population, demographics, and equity indicators within a configurable buffer of your stops and routes. This is useful for Title VI equity analysis, federal grant applications, and answering basic service-planning questions like "how many people live within a quarter-mile of this proposed route?"
